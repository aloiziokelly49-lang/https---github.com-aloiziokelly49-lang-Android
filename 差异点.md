# 答辩功能点 vs 实际代码——差异清单

以下是 `答辩功能点.md`（给老师看的验收点）与 `答辩答案.md`（实际代码实现）之间的 6 处差异。

---

## 差异 1（功能点 1）："SheetViewerScreen" 类名不存在

| | 内容 |
|---|---|
| **答辩点原文** | 文档展示层使用 **SheetViewerScreen** |
| **实际代码** | 使用 `DocumentViewerActivity` + `PdfViewerAdapter` + `RecyclerView`，没有叫 SheetViewerScreen 的类 |

---

## 差异 2（功能点 2）："边界裁剪与透视校正" 未实现

| | 内容 |
|---|---|
| **答辩点原文** | **支持简单的边界裁剪与透视校正** |
| **实际代码** | 只做了图片降采样（限制最大边长 2048px），没有客户端透视校正代码。百度 API 端有 `detect_direction=true` 参数做服务端自动校正 |

---

## 差异 3（功能点 2）："离线提取纯文本" 说法不准确

| | 内容 |
|---|---|
| **答辩点原文** | **系统能离线提取纯文本** |
| **实际代码** | 优先走百度云端 OCR（在线），失败后才降级 Tesseract（离线）。不是纯离线，是"在线优先 + 离线兜底"的双引擎策略 |

---

## 差异 4（功能点 4）："降级搜狗输入法语音" 实际是通用方案

| | 内容 |
|---|---|
| **答辩点原文** | **降级搜狗输入法语音功能** |
| **实际代码** | 用的是通用 `InputMethodManager.showSoftInput()` 弹出键盘，引导用户使用系统键盘自带的语音输入按钮。不绑定搜狗，讯飞/Gboard/搜狗等任何输入法均可 |

---

## 差异 5（功能点 5）："BitmapRegionDecoder" 未使用

| | 内容 |
|---|---|
| **答辩点原文** | **采用分块绘制与 BitmapRegionDecoder 技术** |
| **实际代码** | 用的是"分块渲染（tile rendering）+ Canvas 拼接 + tile.recycle() 即时回收"。没有使用 BitmapRegionDecoder（它适用于解码已有大图，不适用于渲染生成大图的场景） |

---

## 差异 6（功能点 3）："基于开源数学模型" 措辞需要澄清

| | 内容 |
|---|---|
| **答辩点原文** | **基于文字转手写开源数学模型（如 saurabhdaware/text-to-handwriting）** |
| **实际代码** | 算法思路灵感来自该开源项目（高斯随机抖动模拟手写不规则性），但完全是自研的 Android Canvas 渲染引擎，包括逐字绘制、行列排版、多字体加载、笔触系统、纸张纹理等 |
