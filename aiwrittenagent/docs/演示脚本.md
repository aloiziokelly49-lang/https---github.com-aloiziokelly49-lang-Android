## 详细剧本

### 一、开场白（Slide: 项目标题）— 0:00 ~ 0:30

> **（投屏已就绪，展示首页）**

*"各位老师好，我今天演示的项目叫做**云墨 CloudInk**—— 一款基于 Android 原生技术栈开发的一站式多模态手写笔记工具。"*

*"它的核心价值是：将'**文档阅读 + OCR提取 + 语音转写 + 手写排版 + 高清导出**'五件事，在一个 App 内全部完成。全程不依赖任何第三方 PDF 库，OCR 和语音识别全部离线运行。"*

*"接下来我通过一个完整的用户故事串联演示全部核心技术。"*

**（点击首页 → 分屏阅读卡片）**

---

### 二、场景导入: 文档阅读 + 分屏摘录 — 0:30 ~ 2:30

> **（进入分屏阅读页，点击 Toolbar 选择 PDF）**

*"假设我正在阅读一篇学术论文 PDF。左侧使用的是 Android 系统原生的 **PdfRenderer API** 进行逐页渲染，没有任何第三方 PDF 库。"*

*"请大家注意——我可以在中间拖拽这条分割线来调节左右比例，这是完全基于 **ConstraintLayout 的 GestureDetector** 自定义实现的。"*

**（拖拽 SplitBar 左右滑动）**

*"现在我对左侧论文中的某个关键段落做长按摘录——"*

**（长按 PDF 页面 → 弹出摘录对话框 → 输入文字 → 确认）**

*"摘录的文本通过 **EventBus 事件总线** 实时发送到右侧的草稿板。这种架构保证了代码充分解耦——文档模块和摘录模块彼此完全独立。"*

**（点击返回 → 进入手写模拟器）**

---

### 三、核心亮点: 手写渲染引擎参数化 — 2:30 ~ 4:30

> **（进入手写编辑器页面，展示默认渲染效果）**

*"这是本项目的核心技术壁垒——**防 OOM 的手写渲染引擎**。上方的预览画布是基于 Android 原生 Canvas 的自定义 View，没有任何 WebView 参与。"*

*"请看我拖动字距滑块——"*

**（拖动字距滑块，实时预览变化）**

*"注意预览是**所见即所得**的，变化延迟不到 200 毫秒。这是因为我们在 ViewModel 层做了 **Handler 防抖机制**——快速拖动时，只有最后的参数值才会触发后台渲染。"*

*"现在切换笔触——"*

**（点击马克笔）**

*"钢笔、圆珠笔、马克笔，每一种都有独立的笔宽、透明度和阴影参数，模拟真实笔迹的物理特性。"*

*"最关键的是这个**抖动 (Jitter) 参数**——"*

**（拖动抖动滑块从 0 到 1.5）**

*"当抖动为零时，文字像打印机一样整齐排列。当我增大抖动到 1.0 以上时——请注意每个字在 X 轴、Y 轴和旋转角度上产生随机偏移，这正是参考了 GitHub 上开源项目 **text-to-handwriting** 的数学思想，使用 **Gaussian 分布** 来模拟真实手写的不规则感。"*

*"并且，我特别做了**确定性种子**设计——相同的文本和参数组合，渲染结果永远一致，保证预览的稳定性。"*

---

### 四、多模态串联: 语音转写 → 手写渲染 — 4:30 ~ 6:30

> **（滚到下方面板，指向 FAB 录音按钮）**

*"现在展示本项目的第二大技术亮点——**多模态语音协同**。"*

*"点击右下角的录音按钮——"*

**（点击蓝色 FAB，系统弹出录音权限请求 → 授予）**

*"首次使用需要录音权限。"*

**（授予权限，开始说话）**

*"**云墨是一款优秀的笔记工具**"——*

**（说话期间，底部悬浮胶囊面板自动展开，转写文本实时滚动显示）**

*"请大家注意：底部的红色圆点表示正在录音，我说话的文本实时地、逐句地滚动追加到这个胶囊面板中。这是利用了 Android 系统的 **SpeechRecognizer** 引擎，配合 **EXTRA_PARTIAL_RESULTS** 参数，实现了真正的'边录边转写'。"*

*"并且，录音是通过独立的**前台 Service** 实现的——即使我现在按电源键把屏幕关掉，录音和转写也不会中断。"*

**（说话完毕，自动停止）**

*"现在点击**'拼接到草稿'**按钮——"*

**（点击按钮，Toast "已拼接到草稿!"，上方预览画布实时更新）**

*"转写文本被追加到了输入框中，并且触发了手写引擎的渲染管道。从语音到最终手写排版，整个链路完全自动化。"*

---

### 五、技术奠基: OCR 离线识别 — 6:30 ~ 7:00

> **（快速点击返回首页 → 进入 OCR 卡片）**

*"OCR 模块同样是离线运行的——使用的是 **Google ML Kit** 的文字识别引擎。在相机模式下，每一帧通过 **CameraX 的 ImageProxy** 直接传入 ML Kit，无需保存临时图片文件，实现零拷贝识别。"*

*"识别结果同样通过 EventBus 自动流向手写编辑器。"*

**（快速展示拍照/选图，返回首页）**

---

### 六、防 OOM 高潮: 分块渲染 + A4 导出 — 7:00 ~ 9:00

> **（当前在手写编辑器页面，输入或粘贴一大段文本）**

*"现在进入最核心的演示环节——**防 OOM 高清导出**。"*

*"我已经在手写编辑器中录入了一段较长的文本。点击导出按钮——"*

**（触发导出流程：后台渲染 → Loading 转圈 → A4 预览出现）**

*"大家可以看到，系统先将长文本分成了多个独立的 tile 进行渲染，每个 tile 的高度不超过 4096 像素，然后在 Canvas 上合并成一张 **A4 标准尺寸的高清图片**，宽 2480 像素，高 3508 像素，相当于 300 DPI 的印刷级质量。"*

*"这是核心的防 OOM 策略——**分块渲染 + 即时回收**。每画完一个 tile，立刻调用 `Bitmap.recycle()` 释放内存。任何时候只有一张最终大图驻留内存。"*

*"现在我把这张图**保存到系统相册**——"*

**（点击保存，Loading 转圈 → Toast "已保存至 Pictures/CloudInk"）**

*"大家可以在相册 App 中验证——图片完整无损，长文本全部保留。"*

*"再来点击**'导出 PDF'**——"*

**（点击导出 PDF，Toast "PDF 已导出至 Downloads/CloudInk"）**

*"这是用 Android 原生 **PdfDocument** API 生成的 A4 标准 PDF 文件，包含完整的排版内容。"*

---

### 七、历史档案回溯 — 9:00 ~ 9:30

> **（返回首页 → 进入历史档案室）**

*"最后给大家展示一下历史档案回溯功能——"*

**（点击一条记录 → 跳转到手写编辑器且参数恢复）**

*"点击任意历史记录，字距、行距、抖动、纸张、笔触，所有排版参数**完整恢复**。这意味着用户可以随时打开一篇旧笔记继续编辑，达到了'档案可回溯'的体验要求。"*

*"底层数据库使用了 **Room + UUID 主键 + DataStore 偏好存储**，并实现了 v1 到 v2 的增量迁移，保证了数据平滑升级。"*

## 附录: 备用救场话术

### 如果某个功能卡住/崩溃
> *"这个功能在刚才的调试中遇到了设备的兼容性限制，请大家参考我演示录像中的版本。现在让我直接跳到下一个核心模块——"*

### 如果老师追问技术细节
> **OOM 策略**: *"Bitmap内存 = width × height × 4字节。A4 300dpi 约为 35MB。我们的分块策略将单块内存控制在 38MB 以下，同时确保绘制完立刻 recycle。"*
> **EventBus**: *"采用 greenrobot 的 EventBus 3.x，支持 sticky event 和线程模式切换。我们的语音转写使用 ThreadMode.MAIN 确保 LiveData 在主线程更新。"*
> **DataBinding**: *"HandwritingParams 继承 BaseObservable，6 个属性均标注 @Bindable。SeekBar 的 int progress 通过 Activity 的 valueOf/progressOf 方法与 float 参数双向转换。"*

### 如果老师要求看代码
> *"项目已在 GitHub 开源，结构清晰——data 层负责持久化，event 层负责解耦通信，rendering 层负责渲染，ui 层负责视图绑定。我可以现场打开 Android Studio 展示任意模块。"*
