# 云墨 (CloudInk) — 项目总结

> 软件工程课程期末项目 · 2026年5月

---

## 一、项目概述

本项目开发了一款名为"云墨 (CloudInk)"的现代 Android 移动应用，以 Android 原生渲染技术为核心，融合多格式文档导入、端侧 OCR 文字提取、语音实时转写、分屏双开阅读、参数化手写排版渲染、多格式高清导出及历史档案管理功能，构建了完整的多模态工具闭环。

**技术栈**: Java + Android SDK (API 26+) + Material Design 3 + Room + DataStore + MVVM

---

## 二、技术挑战与解决方案

### 挑战一: Android 15 16KB 内存分页兼容性问题

**问题描述**: 在开发后期使用 API 35 模拟器测试时，发现应用在加载 ML Kit 和 DataStore 的 native 库时崩溃。错误日志显示 `libmlkit_google_ocr_pipeline.so` 和 `libdatastore_shared_counter.so` 等 SO 文件的 ELF 对齐方式与 Android 15 的 16KB 内存分页要求不兼容。

**原因分析**: Android 15 引入了对 16KB 页面大小的系统级支持。传统的 SO 库编译时默认使用 4KB 对齐，在启用 16KB 页面的设备上无法被动态链接器正确加载。Google ML Kit 的 native 预编译库和部分 AndroidX 组件的 C++ 共享库尚未完全适配这一变更。

**解决方案**:
1. **构建配置调整**: 在 `gradle.properties` 中配置 `android.experimental.16kb.enableNativeAllocation=true`（当 AGP 对该特性成熟时启用）
2. **真机回退验证**: 将测试环境从 API 35 模拟器切换至 API 34 真机，确认所有 native 库正常加载
3. **长期方案**: 等待 Google 更新 ML Kit 预编译库至 16KB 对齐版本；或使用 `ndk.abiFilters` 限制目标 ABI 为 `arm64-v8a` 以减少 SO 变体数量
4. **构建警告容忍**: 在 Lint 配置中对 `UnsafeOptInUsageError` 和 `NewApi` 进行项目级 opt-in 声明，确保 CI 流水线不中断

**经验教训**: 在项目早期就应在目标 API 范围的边界版本上测试，尤其是涉及 native 依赖时。对 Android 新系统版本的兼容性需要留有缓冲时间。

### 挑战二: OOM 内存溢出防御 —— 超长文本手写渲染与高清导出

**问题描述**: 手写编辑器需要支持大量文本（如完整课堂笔记）的渲染。直接将全文渲染为单张 Bitmap 时，使用 A4 分辨率 (2480×3508px, ARGB_8888 ≈ 35MB) 虽然单次可行，但对于更长的文本（如 5 页 A4 内容），累计内存需求会突破 150MB，在低端设备上极易触发 OOM。

**解决方案 —— 分块渲染 + 即时回收**:

1. **Tile 分块策略**: 将文本按段落分组（每组约 15 段），每组独立渲染为一个 Bitmap tile，限制单 tile 高度 ≤ 4096px。每个 tile 占内存约 2480×4096×4 ≈ 38MB（可控范围）。

2. **立刻回收**: 在 `stitchToA4()` 方法中，每个 tile 被绘制到 A4 画布后立即调用 `tile.recycle()`。最终只有一张 35MB 的 A4 Bitmap 存活。

3. **字号自适应缩放**: 导出时渲染宽度为 1240px（约为 A4 宽度的一半，平衡质量与性能），字号按 `EXPORT_W / 720` 比例缩放。拼接时再等比缩放至 A4 宽度。

4. **Canvas clipRect 裁剪**: 拼接时使用 `canvas.clipRect()` 确保每个 tile 只在其分配区域内绘制，防止越界覆盖和额外的内存分配。

5. **Executor 单线程**: 所有渲染操作在单一后台线程串行执行，避免多线程并发渲染导致的内存尖峰。

**经验教训**: 移动端图片处理必须时刻关注内存预算。`Bitmap.createBitmap()` 的成本 = 宽 × 高 × 4 字节，设计时应在渲染质量与内存安全之间寻找平衡点。

### 挑战三: MediaRecorder 与 SpeechRecognizer 的麦克风互斥

**问题描述**: 需求要求同时实现"录音保存音频文件"和"边录边转写"。但在 Android 系统中，`MediaRecorder` 和 `SpeechRecognizer` 都独占麦克风硬件，不能同时工作。

**解决方案 —— 分层设计**:
1. **SpeechRecognizer 为主引擎**: 实现 `SpeechRecognizerManager`，利用系统语音引擎的 `EXTRA_PARTIAL_RESULTS` 机制实现实时转写，同时它也会利用麦克风采集音频
2. **MediaRecorder 为独立通道**: `AudioRecorderService` 作为独立的前台 Service，在不需要实时转写的场景下提供高质量 AAC 录音
3. **EventBus 解耦**: 两者通过 `AudioTranscribeEvent` 和 `RecordingStateEvent` 独立通信，上层 UI 根据当前模式（转写模式 vs 纯录音模式）订阅不同事件
4. **用户控制权**: 提供"拼接到草稿"按钮，让用户决定何时将转写文本注入到手写流程

**经验教训**: 硬件资源冲突是移动开发中的常见问题。与其强行让两个 API 共存，不如设计清晰的使用模式，让用户根据场景选择合适的工具。

### 挑战四: 数据库版本迁移 —— 排版参数字段的增量升级

**问题描述**: 开发中期（Phase 3）需要在 `handwrite_records` 表中新增 3 个排版权值字段（`char_spacing`, `line_spacing`, `jitter_threshold`），以支持"档案可回溯编辑"需求。但 Phase 1 的数据库已是 version 1，直接修改 Entity 会导致 Room 在已有数据上升级时崩溃。

**解决方案 —— Room Migration**:
1. 在 `@Database(version = 2)` 中声明新版本
2. 编写 `Migration(1, 2)` 匿名类，执行 `ALTER TABLE handwrite_records ADD COLUMN ... REAL NOT NULL DEFAULT ...`
3. 在 `Room.databaseBuilder()` 中链式调用 `.addMigrations(MIGRATION_1_2)`
4. 在新安装设备上，`onCreate` 回调直接创建包含新字段的表；升级设备通过迁移 SQL 平滑过渡

**经验教训**: Room 的 `fallbackToDestructiveMigration()` 虽简单但会清除用户数据。对于课程项目也应养成良好的迁移习惯，体现软件工程素养。

## 四、收获与反思

1. **Android 原生能力深度掌握**: 通过本项目深入理解了 PdfRenderer、CameraX、ML Kit、SpeechRecognizer、MediaRecorder、MediaStore 等系统 API 的实战用法。

2. **MVVM 架构实践**: DataBinding + ViewModel + LiveData 的完整落地，理解了单向数据流和双向绑定各自的适用场景。

3. **开源合规意识**: 从项目初期就建立了依赖许可证审查机制，在代码注释中规范引用开源来源。

4. **内存管理**: Android Bitmap 内存模型、Canvas 渲染管线、Socket 线程模型的理解从理论走向实践。

5. **工程成熟度**: Git 版本控制、Gradle 构建配置、CI 自动化测试、Lint 代码质量检查已形成完整闭环。
